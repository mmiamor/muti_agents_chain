# PM Agent RAG 集成说明

## 概述

PM Agent 现已支持 RAG（Retrieval-Augmented Generation），可以从历史 PRD 库中检索相关知识，辅助生成更高质量的产品需求文档。

## 功能特性

### 1. 自动知识检索
- 当 PM Agent 接收到需求时，自动从数据库中检索相似的历史 PRD
- 基于语义相似度匹配，而非简单的关键词匹配
- 检索结果会作为参考信息添加到系统提示中

### 2. 多维度检索
RAG 系统将每个 PRD 拆分为多个可检索的片段：
- **愿景和目标用户**：产品定位、目标群体、非功能性需求
- **核心功能**：主要功能模块列表
- **用户故事**：每个用户故事的独立片段

这样可以更精确地匹配相关内容。

### 3. 配置灵活
- 可以通过环境变量开关 RAG 功能
- 支持调整检索结果数量（top_k）
- 支持设置相似度阈值（score_threshold）

## 配置说明

### 环境变量

在 `.env.development` 或 `.env.production` 中配置：

```bash
# ── RAG 配置（PM Agent 知识检索）──
ENABLE_RAG_FOR_PM=true          # 是否启用 RAG（默认 true）
RAG_TOP_K=3                     # 返回最相关的 K 个结果
RAG_SCORE_THRESHOLD=0.6         # 相似度阈值（0-1）
RAG_VECTOR_STORE_TYPE=memory    # 向量存储类型（memory 或 chroma）
```

### 配置参数说明

| 参数 | 说明 | 默认值 | 建议范围 |
|------|------|--------|----------|
| `ENABLE_RAG_FOR_PM` | 是否启用 RAG | `true` | - |
| `RAG_TOP_K` | 返回结果数量 | `3` | `1-5` |
| `RAG_SCORE_THRESHOLD` | 相似度阈值 | `0.6` | `0.5-0.8` |
| `RAG_VECTOR_STORE_TYPE` | 向量存储类型 | `memory` | `memory` / `chroma` |

## 向量存储选项

### 1. 内存存储（默认）
- 适合开发和小规模部署
- 数据不持久化，重启后需要重建索引
- 无需额外依赖

### 2. Chroma 存储（生产推荐）
- 持久化存储，支持大规模数据
- 需要安装 `langchain-chroma` 和 `chromadb`
- 更好的性能和可扩展性

安装依赖：
```bash
pip install langchain-chroma chromadb
```

## 工作流程

```
┌─────────────────┐
│  用户输入需求    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ RAG 检索        │ ← 从数据库加载历史 PRD
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 组装上下文       │ ← 历史参考 + 审查反馈
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ PM Agent 生成   │
│ 新的 PRD        │
└─────────────────┘
```

## 数据要求

RAG 系统需要数据库中有已批准的 PRD 才能工作：

1. **PRD 来源**：通过工作流生成的 PRD，经审查员批准后会自动纳入知识库
2. **数据结构**：使用 `Artifact` 表存储，`artifact_type='prd'`，`is_approved=True`
3. **索引更新**：当前实现会在每次检索时从数据库重新加载（后续可优化为增量更新）

## 日志和监控

系统会记录以下日志：

- `[PM Agent] Retrieved similar PRDs from knowledge base` - 成功检索到相关 PRD
- `[PM Agent] No similar PRDs found` - 未找到相关 PRD
- `[PM Agent] RAG retrieval failed: <error>` - RAG 检索失败（不影响主流程）

## 性能考虑

1. **首次加载**：首次检索时会从数据库加载 PRD 并建立索引，可能需要几秒钟
2. **内存占用**：存储在内存中的向量会占用一定内存，取决于 PRD 数量
3. **检索延迟**：每次检索增加约 100-500ms 延迟（取决于数据量和硬件）

## 故障处理

- RAG 检索失败不会阻塞 PM Agent 的主流程
- 如果没有相似 PRD 或 RAG 未启用，PM Agent 仍正常工作
- 日志中会记录所有错误，便于排查问题

## 最佳实践

1. **逐步积累**：随着项目进行，知识库会越来越丰富，检索效果也会提升
2. **定期审查**：定期检查纳入知识库的 PRD 质量
3. **阈值调整**：根据实际情况调整相似度阈值，平衡召回率和精确率
4. **生产环境**：建议使用 Chroma 持久化存储

## 示例场景

### 场景 1：首次使用
- 数据库中没有 PRD
- PM Agent 正常生成 PRD，不使用 RAG
- 该 PRD 被批准后自动纳入知识库

### 场景 2：后续使用
- 数据库中有 10 个已批准的 PRD
- 用户输入"开发一个在线教育平台"
- RAG 检索到 3 个相似的教育类项目 PRD
- PM Agent 参考这些历史 PRD，生成更符合规范的需求文档

### 场景 3：迭代优化
- 用户对 PRD 提出修改意见
- PM Agent 结合审查反馈和历史经验进行优化
- 新版本 PRD 替换旧版本进入知识库

## 未来优化方向

1. **增量索引**：只索引新增或变更的 PRD，而非每次全量加载
2. **混合检索**：结合向量检索和关键词检索，提高召回率
3. **多模态支持**：支持检索设计文档、代码等更多类型的内容
4. **智能过滤**：根据项目类型、时间范围等条件过滤检索结果
5. **缓存机制**：缓存常见查询的检索结果
